# Context Engineering Across Your Data Sources

You have ~10 distinct data sources scattered across `C:\Dev`, several of which are SQLite-backed and several of which are loose files. The pattern across them is consistent enough that one architectural shape would work — and the missing layer is small.

This document is recommendations, not a build plan. It is short on purpose. The principles you already use in `Claudiens/` and `EmeraldTablet/` are the right ones; this just names them and points at the missing piece.

## Why a monolithic database is the wrong answer

It would force a schema migration on every new project. You'd be debating column names instead of building. Worse, it would reduce the *provenance* signal — knowing that a fact came from De Jong's *Alchemical Imagery in the Atalanta Fugiens* is useful precisely because *Claudiens* is a separate, scoped, scholarly artifact. Merging that into a generic `concepts` table strips the provenance.

## What to do instead — federated SQLite with a shared provenance contract

Each source remains its own SQLite file. They never join across files at the schema level. They join at *query time*, via a thin Python layer that knows how to read each one.

The contract is small. Every source agrees on a single shared table named `entities` (or any single canonical name) with this shape:

```sql
CREATE TABLE entities (
    entity_id     TEXT PRIMARY KEY,        -- {db}:{type}:{slug}, e.g. "claudiens:emblem:12"
    canonical_name TEXT NOT NULL,
    entity_type   TEXT NOT NULL,           -- emblem | scholar | concept | mtg_card | tweet | …
    summary       TEXT,                    -- 1–2 sentence description, suitable for LLM context
    source_db     TEXT NOT NULL,           -- which DB this entity lives in
    source_url    TEXT,                    -- where to read more
    confidence    TEXT NOT NULL DEFAULT 'high',  -- high | medium | low
    review_status TEXT NOT NULL DEFAULT 'reviewed',
    updated_at    TEXT NOT NULL
);
```

Each source already has a richer schema underneath; `entities` is just a *view* (or a small materialized table) that exposes a uniform handle the federated layer can index. Writes go to the rich schema; the entities row is generated by a small `seed_entities.py` script that runs after migrations.

## Making it LLM-queryable

Three concrete patterns, in order of leverage:

### 1. SQLite FTS5 on the `summary` column of every entities table
```sql
CREATE VIRTUAL TABLE entities_fts USING fts5(canonical_name, summary, content='entities');
```
Sub-100ms search across 100k+ entities on a laptop. LLMs invoke this via a single tool call.

### 2. Per-database `briefing.md` at the project root
Each project (`Claudiens/`, `EmeraldTablet/`, etc.) maintains a hand-written ~500-word briefing that names: what the database contains, what the columns mean, what the canonical examples are, what's known to be missing. An LLM that reads only this file before querying can write good queries on the first try. You already do this implicitly in `CLAUDE.md` files — formalising it as `briefing.md` (one per DB) makes it portable.

### 3. JSON exports for context windows that don't have file access
For each database, a deterministic `export_for_llm.py` produces:
- `entities.jsonl` — one line per entity (id, name, type, summary)
- `relationships.jsonl` — pairs (entity_a_id, entity_b_id, relation_type)
- `briefing.md` — the same hand-written file as above

These three together fit comfortably in a 1M context window for any single source, and a federated bundle of them fits in 1M for the whole workspace.

## Making it human-queryable

Three patterns:

### 1. CSV exports as first-class artifacts, not afterthoughts
Every `entities` table gets exported as `entities.csv` on each pipeline run. Excel/Numbers/your text editor can grep them.

### 2. A static `index.html` per database
Read-only, no framework, generated from the entities table. List + filter + click-through. You already do this in `Claudiens/site/`. Replicate the pattern for each DB.

### 3. A workspace-level `INDEX.md` listing every database, its briefing path, and its current entity count
One page, refreshed by a script. Makes the workspace navigable without grepping.

## What to NOT do

- **Don't introduce vector embeddings as the primary retrieval mechanism.** SQLite FTS5 on hand-curated summaries beats embeddings for accuracy on this kind of corpus, and it's debuggable. Embeddings are a fallback for when FTS5 misses, not a default.
- **Don't build a single master schema.** Each project's specificity is a feature.
- **Don't skip the briefing files.** They are the highest-leverage artifact in this whole stack — 500 words written once that save thousands of tokens on every future LLM query.
- **Don't move data from project DBs into a workspace DB.** Symlink, federate, query — but don't migrate. Migrating means each scholarly project has to fight the workspace DB's evolution.
- **Don't try to standardise terminology across projects.** *prima materia* in EmeraldTablet and *materia prima* in Claudiens are both correct in their respective contexts; the federated layer should preserve both, and the briefing files should explain the variance. Forcing one canonical form loses provenance.
